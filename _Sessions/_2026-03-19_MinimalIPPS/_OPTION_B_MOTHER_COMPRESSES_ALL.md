# Option B: Mother-Compresses-All (Maximum Context)

**Doc ID**: MIPPS-OPT-B
**Goal**: Reduce ~1MB DevSystem to be usable by cheaper LLMs. Mother (1M context) does ALL compression with full cross-file awareness.
**Status**: Draft - pending user review
**Priority**: Quality over cost

## Core Idea: Single Model Compresses Everything with Full System Context

Mother compresses every file with all 104 files in cached context. Zero cross-file context loss. No delegation to cheaper models.

## Source and Scope

**Source directory**: `.windsurf/` (active DevSystem)

**File inventory** (verified 2026-03-19):
- Rules: 8 md
- Workflows: 36 md
- Skill docs (SKILL.md + supporting): 24 md
- Skill prompts: 7 md
- Python scripts: 20 py (skip)
- JSON configs: 9 json (skip)
- **Compressible: 75 files** | Non-compressible: 29 files (copy as-is)

## Why Higher Quality Than Option A

Option A's Transformer compresses `build.md` without seeing that `edird-phase-planning.md` and `devsystem-core.md` reference concepts defined in it. Mother has the entire system in context - it knows every cross-file dependency while compressing.

- **Zero context loss** during compression
- **Consistent voice** across all compressed files (one model, one style)
- **Cross-file concept preservation** guaranteed (Mother sees all references)

## Step-to-Output Mapping

Aligned with NOTES.md Steps 1-7:

- **Step 1**: Read all files → `context/all_files_bundle.md` (bundle command)
- **Step 2**: Analyze call tree → `_01_FILE_CALL_TREE.md` (Mother)
- **Step 3**: Analyze complexity → `_02_FILE_COMPLEXITY_MAP.md` (Mother)
- **Step 4**: Compression strategy → `_03_FILE_COMPRESSION_STRATEGY.md` (Mother)
- **Step 5**: Generate guidelines → `prompts/transform/` + `prompts/eval/` (Mother creates type-specific guidelines for its own use in Step 6)
- **Step 6**: Compress files → `output/` (Mother compresses, Verification judges)
- **Step 7**: Verify results → `_04_FILE_COMPRESSION_REPORT.md` (Verification model)
- **Step 8**: Functional test → run compressed system on target LLM (optional)
- **Iterate**: Mother reviews report, re-compresses failed files

## Pipeline Commands

```

mipps_pipeline.py check     --step 2|3|4                  → Verify Mother output via Verification model
```

## Model Roles

1. **Mother** - Claude Opus 4.6 Thinking (1M context)
   - Steps 2-7: ALL analysis and compression
   - Cached context reused for every call
   - Produces consistent output across all files
   - **Cache risk**: Anthropic cache TTL is 5 minutes. 75 sequential calls at ~30s each = ~38 minutes. Cache should survive, but monitor for expiration. If cache expires, re-send bundle as first message to re-warm.

2. **Verification** - GPT-5-mini or Claude Haiku 4.5
   - Step 6: Judge each compressed file against eval/ prompt
   - Step 7: Compare original vs compressed, produce per-file report
   - `check` command: Spot-check Mother outputs, feed issues back

## Compression Flow (Step 6 Detail)

For each of the 75 compressible files:

```
Mother call (cached 300K context):
  Input: "Compress this file according to the strategy.
          File type: [rules|workflow|skill_doc|...]
          Compression guidelines: [from transform/ file for this type, generated in Step 5]
          Strategy excerpt: [Primary/Secondary/Drop lists for this file]
          Current file content: [full file]"
  Output: Compressed version

Verification judge call:
  Input: original + compressed + eval/ prompt
  Output: score 1-5 + issues

If score < threshold (3.5):
  Mother refinement call (cached context):
    Input: "Refine this compression. Judge feedback: [issues]"
    Output: Improved version
  Re-judge

Save best version to output/
```

## Step 7 Report Format

## Cost Estimates

**Base assumptions**: ~300K tokens cached context, ~75 compressible files
**Pricing**: Opus 4.6: $5.00/1M in, $0.50/1M cached, $25.00/1M out (includes thinking)

**Steps 2-4: Mother analysis**
- Call 1 (call tree): 300K × $5/1M + ~80K out × $25/1M = $1.50 + $2.00 = $3.50
- Call 2 (complexity, cached): $0.15 + $2.25 = $2.40
- Call 3 (strategy, cached): $0.15 + $1.75 = $1.90
- **Steps 2-4: ~$7.80**

**Step 5: Mother generates prompts (cached)**
- ~5 calls × ($0.15 + $0.50) = ~$3.25

**Step 6: Mother compresses (cached)**
- Per file: 300K cached × $0.50/1M + ~10K out × $25/1M = $0.15 + $0.25 = ~$0.40
- 75 files × $0.40 = ~$30.00
- Judge (Verification): 75 × $0.004 = ~$0.30
- Refinement (~20% fail): 15 × $0.40 = ~$6.00
- **Step 6: ~$36.30**

**Step 7: Verification**
- 75 × $0.005 = ~$0.38

**Iteration (Mother, cached): ~$2.50**
- Mother reads report, identifies files needing re-compression
- Re-compresses only failed files (typically 5-10)
- Re-runs verification for changed files

**First iteration: ~$50.25**
**Subsequent iterations: ~$39.20** (skip Steps 2-4, re-compress only failed files)

## Cost Optimization Levers

- **Selective compression**: Only compress files flagged as Secondary/Drop in strategy (~40 files instead of 75) → ~$28 first iteration
- **Batch judge calls**: 50% discount via batch API → saves ~$0.15
- **Skip refinement**: Accept first pass, iterate at Step 7 level → saves ~$6.00

## Strengths

- **Maximum cross-file awareness** - Mother sees all 104 files while compressing each
- **Consistent voice** - single model, no style drift between compressed files
- **Self-referencing analysis** - Mother references its own call tree and complexity map during compression
- **No concept orphaning** - if concept X is dropped from file A, Mother knows to drop references in files B and C

## Weaknesses

- **5x cost of Option A** (~$50 vs ~$16.50)
- **Sequential compression** - each file waits for Mother (no parallel, ~75 API calls)
- **Mother may be too conservative** - deep understanding may make it reluctant to drop things
- **No functional validation** without Step 8 (same gap as Option A)

## Open Questions

1. **Compression ordering**: Compress rules first, then workflows, then skills? Full context may make ordering irrelevant, but earlier outputs could inform later compressions.
2. **Quality degradation**: Does output quality degrade after 75 sequential calls? Consider batching by file type with fresh prompts if degradation observed.
3. **Functional testing**: Add Step 8 with target LLM tests? See Option D for test-driven approach - could combine B's compression quality with D's validation.
