# Option C: Dependency-Ordered Compression with Propagated Context

**Doc ID**: MIPPS-OPT-C
**Goal**: Reduce ~1MB DevSystem to be usable by cheaper LLMs. Compress in dependency order so each file sees what survived in upstream files.
**Status**: Draft - pending user review
**Priority**: Quality over cost (highest quality, highest cost)

## Core Idea: Layered Compression Where Each File Sees What Survived Upstream

Build semantic dependency graph. Compress in topological order: roots first (rules, no dependencies), leaves last (workflows, depend on everything). When compressing any file, include *already-compressed* versions of all upstream files. The compressing model sees exactly what survived - broken cross-file references become impossible.

## Source and Scope

**Source directory**: `.windsurf/` (active DevSystem)

**File inventory** (verified 2026-03-19):

- Skill prompts: 7 md
- Python scripts: 20 py (skip)
- JSON configs: 9 json (skip)

**Exclusion criteria** (skip compression):
- Files < 100 lines AND rarely loaded → copy as-is
- Applies to: infrequently-invoked workflows, supporting skill docs (SETUP.md, UNINSTALL.md)
- Reduces compression scope by ~20-30%

## Why Higher Quality Than Option A

Option A compresses files independently. File A drops concept X. File B references concept X and keeps the reference - broken link. Option C prevents this: when compressing File B, the Transformer sees compressed File A and knows concept X is gone.

Option B (Mother-All) also has full context, but Mother may preserve too much because it sees everything. Option C forces explicit decisions: "concept X was dropped in Layer 0, so Layer 1 must adapt."

## Model Roles

1. **Mother** - Claude Opus 4.6 Thinking (1M context)
   - Steps 2-4: Analysis with full cached context
   - Step 4b: Build semantic dependency graph (concept-level, not just file-level)
   - Iteration: Review report, update strategy, identify cascade effects
   - **Cache note**: 4 calls at ~30s each = ~2 minutes. Cache TTL (5 min) is not a concern here.

2. **Sonnet** - Claude Sonnet 4.5 Thinking (200K context)
   - Step 5: Generate layer-aware transform/ + eval/ prompts
   - Step 6: Compress files (200K context fits layer context + file)
   - Quality justification: Sonnet follows complex instructions better than GPT-5-mini, critical for maintaining concept coherence across layers

3. **Verification** - GPT-5-mini or Claude Haiku 4.5
   - Step 6: Judge each compressed file
   - Step 7: Cross-file consistency check (reads ALL compressed files together)

## Compression Flow (Step 6 Detail)

### Layer 0: Rules (most critical, compressed first)

```
For each rule file (8 files):
  Sonnet call:
    Context: compression strategy excerpt + compress_rules.md prompt
    Input: original file
    Output: compressed version (3 candidates)
  Judge: score each candidate, select best ≥ threshold
  Refine if needed

After all 8 rules compressed:
  Generate layer0_compressed.md (concatenated compressed rules)
  Lock Layer 0: set pipeline_state.json layer0_locked=true, no changes until iterate command
```

### Layer 1: Skill Docs (see compressed rules)

```
For each skill doc (24 files):
  Sonnet call:
    Context: layer0_compressed.md + compression strategy + compress_skill_docs.md
    Input: original file
    Output: compressed version (3 candidates)
  Judge: score, select best
  Refine if needed

Sonnet sees compressed rules → adapts references automatically.
If compressed rules dropped "Temporary Files" concept, Sonnet removes
references to it from skill docs.

Generate layer1_compressed.md, lock Layer 1 (layer1_locked=true).
```

### Layer 2-3: Same pattern, accumulating upstream compressed context.

Layer 3 (workflows) context = layer0 + layer1 + layer2 compressed (~30-50K tokens).
Well within Sonnet's 200K window.

## Cost Optimization Levers

- **Use Sonnet for Layer 0 only, GPT-5-mini for Layers 1-3**: Drops Step 6 to ~$25 total
- **Reduce candidates to 2**: Saves ~$27
- **Batch API**: 50% discount on judge calls
- **Skip Layer 2** (skill prompts are domain-specific, copy as-is): Saves ~$7

## Strengths

- **Zero broken cross-file references** - each file sees what survived upstream
- **Layered quality** - rules (most critical) get most attention
- **Explicit cascade tracking** - dependency graph documents exactly what affects what
- **Dependency graph is reusable** - valuable beyond compression project

## Weaknesses

- **Most expensive** - ~$96 first iteration (6x Option A)
- **Cascade failure risk** - bad Layer 0 compression propagates to all layers
- **Layer boundaries may be imprecise** - some circular dependencies exist
- **Sequential within layers** - Layer 1 cannot start until Layer 0 is locked
- **No functional validation** without Step 8

## Open Questions

1. **Circular dependencies within Layer 3**: Workflows like `/go` → `/recap` → `/continue` reference each other. Mitigation: compress in alphabetical order within layer; each workflow sees previously-compressed peers. Not perfect but limits breakage.
2. **Layer 0 quality gate**: Add Mother review of Layer 0 output before Layer 1? Adds ~$2.50 but prevents cascade failures. Recommended for first iteration.
3. **Hybrid with Option D**: Add functional tests after Layer 3? Combines C's reference integrity with D's behavioral validation. See Option D for test suite design.
